# 项目理解

## 1. 项目定位
- 这是一个 Go 语言实现的“AI 外卖助手”项目，目标是用聊天式交互完成推荐与自动下单。
- 当前仓库形态是“网关单体可运行 + 多服务代码骨架并存”。
- 实际可直接运行并对外提供能力的是 `api-gateway/main.go`。

## 2. 当前可运行主链路
- 启动入口：
  - `run.bat` -> `scripts/run.ps1`
  - 静默启动：`run_silent.vbs`
  - 停止：`stop.bat` / `stop_silent.vbs` -> `scripts/stop.ps1`
- `scripts/run.ps1` 的行为：
  - 自动准备 Go 环境（系统 Go 或下载便携 Go）
  - 注入运行环境变量（HTTP/TCP/UDP、MySQL、Redis、ALLOWED_HOST）
  - 执行 `go mod tidy`
  - 启动 `go run ./api-gateway`
  - 等待 HTTP 就绪并做 smoke test（注册/登录/聊天/自动下单/TCP PING/UDP PING）

## 3. 运行时架构（真实代码）
- `api-gateway/main.go`
  - 同时启动三种服务端口：
    - HTTP：Gin Web/API（默认 `8080`）
    - TCP：文本请求推荐（默认 `9091`）
    - UDP：文本请求推荐（默认 `9092`）
  - 提供静态前端：`web/index.html` + `web/app.js` + `web/app.css`
  - 主要接口组：
    - 用户：注册、登录、鉴权、偏好问卷读写
    - 订单：创建、查询、自动下单支付（当前为内存模拟）
    - 推荐/聊天：根据输入关键词返回推荐商家
    - 网络：返回 HTTP/TCP/UDP 地址

## 4. 核心业务流程
- 用户流程
  - 前端先登录/注册，拿到 Bearer token。
  - token 保存在本地存储，后续请求带 `Authorization`。
- 偏好流程
  - 首次登录后获取问卷题目，逐步提交。
  - 偏好写入 MySQL，缓存到 Redis（可用时）。
- 推荐流程
  - 聊天接口 `/api/v1/chat/send` 接收自然语言需求。
  - 服务端按关键词（如“轻食/减脂/辣/川菜”）走规则推荐，返回商家列表与回复文案。
- 下单流程
  - 前端点击商家“自动下单支付”。
  - 网关直接在内存 `orderStore` 里创建“已支付”订单并返回结果。

## 5. 数据与存储
- MySQL（`pkg/database/mysql.go`）
  - 当前网关初始化时会 `AutoMigrate`：
    - `users`
    - `user_preferences`
  - `scripts/init_db.sql` 同时给出另一套分表方案样例（`user_preferences_0`、`orders_202601`）。
- Redis（`pkg/cache/redis.go`）
  - 用作 JSON 缓存与会话缓存。
  - Redis 不可用时，网关自动降级为 DB/内存路径。
- 内存数据
  - `sessions`：登录 token 映射
  - `orderStore`：订单数据（重启丢失）

## 6. 目录与模块状态
- 已落地并被主流程调用
  - `api-gateway/`
  - `pkg/cache`
  - `pkg/database`
  - `web/`
  - `scripts/run.ps1`、`scripts/stop.ps1`
- 有代码但当前未接入主运行链路（偏“预研/骨架”）
  - `services/*`：用户、订单、推荐、支付、爬虫、AI 客户端等
  - `pkg/mq`、`pkg/metrics`、`pkg/lock`、`pkg/concurrency`、`pkg/middleware`
  - `proto/*.proto`（定义了 gRPC 协议，但当前未见服务实现和注册）
- 空目录
  - `services/notification-service`
  - `cmd/`

## 7. 与文档声明的差异（重要）
- 文档描述的是“8+ 微服务 + gRPC + Kafka + Redis Cluster + 完整治理”目标架构。
- 实际可运行实现目前更接近“单网关应用”：
  - 推荐和自动支付主要是规则/模拟逻辑；
  - 订单未落库（主链路走内存）；
  - 多服务、Kafka 流、分布式事务、Jaeger 实际链路尚未打通。

## 8. 外部依赖与环境
- 代码依赖：Gin、GORM、Redis、Kafka、Playwright、Prometheus。
- 基础设施编排：`scripts/docker-compose.yml`
  - MySQL、Redis、Kafka、Consul、Prometheus、Grafana、Jaeger。
- Windows 暴露公网辅助：`scripts/expose-public.ps1`（开放防火墙并打印访问地址）。

## 9. 当前项目结论
- 这是一个“目标架构完整、落地程度分层”的项目：
  - 用户可直接启动并体验前端聊天推荐与自动下单演示；
  - 同时保留了向微服务演进所需的大量模块雏形。
